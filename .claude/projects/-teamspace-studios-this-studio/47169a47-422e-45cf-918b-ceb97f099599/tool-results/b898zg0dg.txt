This is a designed behavior, not a bug, but should be documented explicitly.\n\n---\n\n## 1.5 Attention Mechanisms\n\nThe `MultiTypeEQParameterHead` already incorporates a dual-path frequency localization mechanism: a local CNN (`mel_cnn`, kernel sizes 7 and 5), a dilated CNN (`mel_cnn_dilated`, dilation 4 and 8), a merge layer, input-conditioned queries (`query_proj`), Gaussian position priors (`attn_position_bias`), and a learnable blend with a direct regression path (`freq_blend_weight`). This is already a sophisticated attention-like mechanism. The `attn_temperature=0.1` sharpens the softmax heavily.\n\nAdding a full Transformer self-attention layer (Audio Spectrogram Transformer, Gong et al., 2021, Interspeech) on top of this existing mechanism risks destabilizing the attention training dynamics rather than improving them. The recommendation is to stabilize the existing dual-path attention before adding cross-attention layers.\n\nThe `freq_blend_weight` parameter (`nn.Parameter(torch.tensor(0.0))`) — initializing `sigmoid(0)=0.5` — is a correct design: equal blend prevents either path from being zeroed at initialization. Monitor `alpha = sigmoid(freq_blend_weight)` during training to verify both paths are actually being used.\n\n---\n\n## 1.6 Layer Sizing and Initialization\n\nThe code contains explicit small-weight initialization for the output-adjacent layers:\n```python\nnn.init.xavier_uniform_(self.trunk.weight, gain=0.5)   # half-amplitude Xavier\nnn.init.xavier_uniform_(self.gainq_head.weight, gain=0.1)  # very small init\nnn.init.zeros_(self.gainq_head.bias)\n```\n\nWith `gain=0.1` on `gainq_head`, the initial output before `tanh` (for gain) will be near zero, so `tanh(~0) ≈ 0` → initial gain ≈ 0 dB. The `sigmoid` path for Q similarly: with small weights, sigmoid output ≈ 0.5 → Q ≈ exp(0.5*(log10 - log0.1) + log0.1) ≈ exp(0.5*4.6 - 2.3) ≈ exp(0) = 1.0. Both are in the linear regime of their activations. Initial output saturation is not occurring.\n\nThe `CausalConv1d` and `GatedResidualBlock` layers use default PyTorch initialization (Kaiming uniform for `nn.Conv1d` since PyTorch 1.0+), which is appropriate for the tanh×sigmoid gating. If gradients are still exploding (NaN loss), reduce to 0.5.\n- Clipping should be applied before every optimizer step, after `loss.backward()`.\n\n**Connection to the observed metrics.** The fact that train_loss is 32.65 while val_loss is 16.19 may partly reflect gradient explosions during training: the optimizer overshoots minima repeatedly in the training set (raising the effective training loss) while the evaluation pass (no gradient, no update) measures a smoother loss surface. Gradient clipping would reduce this effect.\n\n**Citations.**\n- Pascanu, Mikolov & Bengio (2013). \"On the Difficulty of Training Recurrent Neural Networks.\" ICML. Issues relevant to this model:\n\n- At high-loss starting points, the gradient magnitudes are large relative to what `sqrt(v_t)` can track, causing large steps that may overshoot. The first few steps are particularly aggressive.\n- L2 regularization in Adam is absorbed into the adaptive update (as described in 3.1), making effective weight decay weaker than intended.\n- With `epsilon=1e-8`, Adam is numerically sensitive to very small gradient variances (can cause overly large updates in low-gradient-variance parameters — common in bias terms and normalization parameters).\n\n**Tuning epsilon.** For high-magnitude loss tasks, increasing epsilon to `1e-7` or `1e-6` can stabilize early training by reducing step sizes when second-moment estimates are small.\n\n**Citations.**\n- Kingma & Ba (2015). \"Adam: A Method for Stochastic Optimization.\" ICLR.\n\n---\n\n### 4.2 AdamW (currently in use)\n\nAdamW (Loshchilov & Hutter, 2019, ICLR) corrects Adam's L2 regularization flaw as described in 3.1. It is strictly superior to Adam for tasks where weight decay is intended to be a true L2 regularizer.\n\n**Correct weight_decay values.** See 3.1: `1e-4` for this model given current high train_loss, adjustable upward once the train/val relationship normalizes.\n\n**Recommended configuration for this model:**\n- `lr = 3e-4` (with warmup from 1e-6)\n- `betas = (0.9, 0.999)`\n- `epsilon = 1e-8` (default; increase to `1e-7` if early instability occurs)\n- `weight_decay = 1e-4`\n\n**Citations.**\n- Loshchilov & Hutter (2019). \"Decoupled Weight Decay Regularization.\" ICLR.\n\n---\n\n### 4.3 RAdam (Rectified Adam)\n\n**Mechanism.** Liu et al. (2020, \"On the Variance of the Adaptive Learning Rate and Beyond,\" ICLR) identified that in early Adam training, the adaptive learning rate has very high variance because the second-moment estimate `v_t` is unreliable. RAdam computes a rectified step: when the estimated effective rank of the second-moment estimate is below a threshold (`rho_inf - 4`), RAdam falls back to SGD with the current LR (bypassing the adaptive scaling entirely). When above the threshold, it applies a corrected adaptive update.\n\n**Relevance when train_loss is very high early.** RAdam is particularly valuable in the first 5–15% of training when `v_t` is poorly estimated. It prevents the extreme early-step overshoots that Adam/AdamW can produce, effectively achieving the same stabilization goal as LR warmup without requiring warmup hyperparameter tuning. Liu et al. (2020) show RAdam matches or exceeds AdamW across NLP, CV, and speech tasks in both convergence speed and final accuracy.\n\n**When to prefer RAdam over AdamW + warmup.** If the warmup hyperparameters are difficult to tune (e.g., it is unclear whether 500 or 2000 warmup steps is correct), RAdam with its principled early-step correction is a more robust choice. For this model with high initial loss, RAdam is a strong candidate.\n\n**PyTorch.** Available as `torch.optim.RAdam` since PyTorch 1.10.\n\n**Citations.**\n- Liu et al. (2020). \"On the Variance of the Adaptive Learning Rate and Beyond.\" ICLR.\n\n---\n\n### 4.4 AdamW + Lookahead\n\n**Mechanism.** Zhang et al. (2019, \"Lookahead Optimizer: k steps forward, 1 step back,\" NeurIPS) wraps any optimizer with a slow-weight update: the inner optimizer (AdamW) takes k fast steps, then the slow weights are updated as a linear interpolation: `phi = phi + alpha * (theta - phi)`. Default values: k=5, alpha=0.5.\n\n**Stability benefits.** Lookahead reduces the variance of the inner optimizer's updates, which is particularly useful when the loss landscape has sharp, narrow valleys (common in multi-task losses combining regression and classification). Zhang et al. show Lookahead converges faster and more stably than the inner optimizer alone on NLP and CV benchmarks.\n\n**Practical consideration.** Lookahead is not natively in PyTorch and requires a wrapper implementation (available in `timm` library and as a standalone package). The added complexity may not be warranted unless AdamW alone shows instability.\n\n**Citations.**\n- Zhang et al. (2019). \"Lookahead Optimizer: k steps forward, 1 step back.\" NeurIPS.\n\n---\n\n### 4.5 Lion\n\n**Mechanism.** Chen et al. (2023, \"Symbolic Discovery of Optimization Algorithms,\" NeurIPS) discovered the Lion optimizer via program search. It uses only the sign of the gradient update (no second-moment estimate), applying `theta = theta - lr * (sign(beta1 * m + (1-beta1) * g) + wd * theta)`. This gives uniform step sizes across all parameters, modulated only by sign.\n\n**Memory efficiency.** Lion maintains only the first-moment (m), using half the optimizer state memory of Adam/AdamW (which also stores v). For large TCN models with many channels, this can be significant.\n\n**Convergence characteristics.** Chen et al. report Lion outperforms AdamW on image classification and language modeling at the same compute budget, particularly at larger batch sizes. The sign-update property means Lion effectively uses a constant step size, making it sensitive to LR magnitude — the recommended LR is 3–10× smaller than AdamW's LR for the same model.\n\n**Cautions for this use case.** Lion's uniform step size can be problematic for multi-task losses where different loss components have very different gradient scales (here: Huber regression ~O(1–10) vs. cross-entropy classification ~O(0.1–1) vs. frequency response L1 ~O(?)). The adaptive scaling in AdamW explicitly handles these scale mismatches; Lion does not. [EMPIRICAL — limited direct audio TCN citations for Lion.]\n\n**Citations.**\n- Chen et al. (2023). \"Symbolic Discovery of Optimization Algorithms.\" NeurIPS.\n\n---\n\n### Recommendation: Given the 2× Train/Val Loss Ratio\n\nThe inverted ratio (train >> val) combined with high absolute train_loss is most consistent with one of:\n- **Gradient explosion / instability in training steps** (fixed by gradient clipping + warmup).\n- **Training/validation distribution mismatch** (fixed by data splitting strategy; see Section 5.1).\n- **Loss computed differently during training vs. validation** (e.g., Gumbel-Softmax temperature active in training, hard argmax in validation; check that the loss function behaves identically).\n\n**Optimizer recommendation.** Use RAdam with `lr=3e-4, weight_decay=1e-4` as a first experiment. A naive implementation that uses `torch.rand` or `random` without seed control will generate different distributions in training and validation depending on initialization order, worker count, and epoch count — and may not generate truly non-overlapping samples.\n\n**The inverted loss ratio as a data leakage signal.** The most probable cause of val_loss being half of train_loss in a synthetic on-the-fly system is that the validation generator uses a different (easier) EQ configuration distribution than the training generator. For example:\n- If validation uses fewer active bands, smaller gain ranges, or less extreme Q values, the val_loss will naturally be lower not because the model generalizes well, but because the task is easier.\n- If the training generator applies more complex configurations (e.g., multiple HP/LP filters combined) while validation uses primarily peaking EQ, the classification cross-entropy component of the loss will be harder during training.\n\n**Best practices for on-the-fly datasets.**\n- Assign fixed random seeds per split: training seed = 42, validation seed = 99. Use `torch.Generator` with explicit seeding per DataLoader worker (via `worker_init_fn`) to ensure reproducible, non-overlapping sequences.\n- Identical configuration distributions: ensure training and validation DataLoaders call `SyntheticEQDataset` with the same parameter ranges (gain, freq, Q ranges; filter type probabilities).\n- Use index-based seeding: if the dataset can be indexed, derive the per-sample seed from the global seed and index, ensuring training indices [0, N) and validation indices [N, N+M) produce non-overlapping samples.\n\n**PyTorch DataLoader worker seeding.** The official PyTorch documentation recommends setting `worker_init_fn = lambda worker_id: np.random.seed(base_seed + worker_id)` to prevent all workers from using identical random streams. Without this, multiple workers may generate identical samples.\n\n**Citations.**\n- PyTorch Documentation: \"Reproducibility\" (torch.manual_seed, DataLoader.worker_init_fn).\n\n---\n\n### 5.2 Class Imbalance in Filter Type Classification\n\n**The imbalance risk.** The 5 filter types (peaking, lowshelf, highshelf, highpass, lowpass) in `SyntheticEQDataset` are unlikely to be uniformly generated in practice: peaking EQ is the most common in real-world audio mixing, so synthetic datasets that model realistic distributions will over-represent peaking relative to highpass/lowpass. Even with uniform generation, the cross-entropy component of `MultiTypeEQLoss` will show imbalanced per-class accuracy.\n\n**Detection.** Compute per-class accuracy (or recall) for each of the 5 filter types on the validation set. A class with < 50% accuracy while the majority class has > 80% indicates imbalance-driven degradation.\n\n**Fix 1: Class weighting.** Pass `weight` tensor to `torch.nn.CrossEntropyLoss` with inverse-frequency weights: `weight[c] = N_total / (N_classes * N_c)`. This is standard practice and is well-supported in PyTorch.\n\n**Fix 2: Focal Loss.** Lin et al. (2017, \"Focal Loss for Dense Object Detection,\" ICCV) modulates cross-entropy by `(1 - p_t)^gamma` where p_t is the model's predicted probability for the correct class. This down-weights well-classified examples (majority peaking class) and focuses learning on the hard/rare classes (highpass, lowpass). For `gamma=2` (the default in the original paper), focal loss effectively increases the relative weight of rare filter types during training. Given that `MultiTypeEQLoss` already includes cross-entropy and the Hungarian matching is solving a permutation problem, focal loss on the type classification component is a targeted fix for per-class imbalance.\n\n**Fix 3: Label Smoothing.** Szegedy et al. (2016, \"Rethinking the Inception Architecture,\" CVPR) showed that replacing one-hot targets with soft targets (e.g., true class = 0.9, others = 0.1/4 each) improves calibration and reduces overconfidence. For the filter type head, label smoothing of `eps=0.1` is a low-cost regularizer that also reduces the model's overconfidence in the majority class.\n\n**Fix 4: Balanced Synthetic Generation.** The simplest fix: enforce uniform filter type sampling in `SyntheticEQDataset` by design, rather than allowing natural (potentially skewed) sampling. This eliminates the imbalance at the source.\n\n**Citations.**\n- Lin et al. (2017). \"Focal Loss for Dense Object Detection.\" ICCV.\n- Szegedy et al. PyTorch: accumulate gradients with `loss.backward()` over k steps, then `optimizer.step(); optimizer.zero_grad()`.\n\n**For large effective batch training on constrained hardware.** If target batch size is 64 but GPU memory allows only 8 samples, use `accumulation_steps = 8`.
