### 1. The "Fat Head" & Underfitting Fix (Architecture)
A Fully Connected head with 27.5M parameters fed by a CNN is a classic "hourglass" bottleneck. The CNN is doing too little work, and the FC head is overwhelmed by raw, high-dimensional data ($53,376$ inputs). 

**The Fix:** We need to force the CNN to compress the spatial dimensions further before handing it off to the FC layers. We can do this cleanly by adding a single `nn.AdaptiveAvgPool2d` right before the flatten operation.

```python
# In model_cnn.py
self.cnn = nn.Sequential(
    # ... your existing Conv2d blocks ...
    nn.Conv2d(128, 256, 3, padding=1), 
    nn.BatchNorm2d(256), 
    nn.ReLU(), 
    nn.MaxPool2d(2),
    # Add this to crush the spatial dimensions to 2x4 before flattening
    nn.AdaptiveAvgPool2d((2, 4)) 
)

# 256 channels * 2 * 4 = 2048
self.flatten_dim = 2048 

self.fc = nn.Sequential(
    nn.Linear(self.flatten_dim + bypass_dim, 512), # Now ~1M params instead of 27M
    nn.ReLU(),
    nn.Dropout(0.3),
    nn.Linear(512, 256),
    nn.ReLU(),
    nn.Linear(256, 24)
)
```

### 2. The Input Normalization Crisis
Feeding raw Mel-spectrograms (which typically range from $-80\text{dB}$ to $+20\text{dB}$) into an unnormalized CNN guarantees severe gradient variance. However, because we established previously that `BatchNorm2d` destroys the absolute amplitude of your difference channel, **do not use BatchNorm on the input.**

**The Fix:** Use a fixed global scaler in your `Dataset` or `DataModule` so the absolute dB relationships are perfectly preserved, but the neural network receives data in a sane `[-1, 1]` or `[0, 1]` range.

```python
# In your dataset or dataloader script
def __getitem__(self, idx):
    stack = np.load(self.files[idx])['mel_stack']
    # Assuming typical mel bounds of -80dB to +20dB
    # Shift to 0, then scale. 
    stack = (stack + 80.0) / 100.0 
    
    # ... return tensor
```

### 3. The Q-Range Mismatch
If your target generator is creating Q-factors between `[0.5, 4.0]`, but your normalization function divides by `10.0`, the neural network is only predicting in a tiny window of `[0.05, 0.4]`. This creates artificially small gradients for Q.

**The Fix:** Align the constants universally.
```python
# In whatever config file or script holds these globals
MIN_Q = 0.5
MAX_Q = 4.0

# In your label normalization
def normalize_q(q_val):
    return (q_val - MIN_Q) / (MAX_Q - MIN_Q)
```

### 4. Optimizer & Scheduler Adjustments
Audio loss landscapes are notoriously noisy. `Adam` without weight decay allows weights to grow unchecked, and a `patience=3` on the scheduler forces the learning rate down at the first sign of standard batch noise.

**The Fix:** Switch to `AdamW` (Adam with proper Weight Decay) to naturally regularize the network, and give the scheduler room to breathe.

```python
# In your PyTorch Lightning Module
def configure_optimizers(self):
    # Add weight_decay to gently penalize massive weights
    optimizer = torch.optim.AdamW(self.parameters(), lr=3e-4, weight_decay=1e-4)
    
    scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau(
        optimizer, 
        mode='min', 
        factor=0.5, 
        patience=6, # Doubled from 3
        min_lr=1e-6
    )
    
    return {
        "optimizer": optimizer,
        "lr_scheduler": {
            "scheduler": scheduler,
            "monitor": "val_loss"
        }
    }
```
