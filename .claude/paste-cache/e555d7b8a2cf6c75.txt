The main issue was model_tcn.py:264 had n_mels=0 hardcoded, which completely disabled the attention-based
    frequency estimation mechanism in MultiTypeEQParameterHead. The model was falling back to a simple sigmoid
    regression path that produces blurry, imprecise frequency estimates.

    This single line meant:
    - The attention mechanism over mel-spectrogram bins (which localizes EQ peaks) was never activated
    - freq_attn, pos_embed, mel_cnn, query_proj — all these learned parameters were unused
    - Frequency was predicted solely via a single linear layer → sigmoid mapping, which can only produce coarse
    frequency estimates

    This explains why freq_mae was stuck at ~2 octaves (barely better than random over the 10-octave range) even
     though train_loss was decreasing — the model was learning gain/Q and spectral shape but had no pathway to
    learn precise frequency positions.

    The secondary issues (attention entropy collapse from -1.5 position bias, Huber delta too large,
    lambda_freq_anchor too low) were also real but would have been masked by this primary bug.
