# Overfitting & Training Instability: Diagnostic & Remediation Plan

## Executive Summary

**This is NOT a simple overfitting problem.** The training log (`training_v8.log`) reveals a **catastrophic numerical collapse** starting at epoch 5, where validation loss becomes `NaN` and training loss drops to `0.0000`. The train/val gap observed in epochs 1-4 (22.65→14.73 train vs 21.34→14.64 val) is the *prelude* to the actual failure.

### Root Cause Diagnosis (Priority Order)

| # | Root Cause | Severity | Evidence |
|---|-----------|----------|----------|
| 1 | **NaN gradient explosion** in biquad DSP layer | CRITICAL | val_loss=NaN from epoch 5, q_mae=NaN, train=0.0 |
| 2 | **Loss weight imbalance** causing gradient conflict | HIGH | loss_gain dominates (~5-9), spread_loss negative, embed_var_loss=0 |
| 3 | **Insufficient regularization** for model capacity | MEDIUM-HIGH | 1.8M params on 10K samples, weight_decay=1e-5 too low |
| 4 | **Gumbel-Softmax temperature too aggressive** | MEDIUM | tau=2.0→1.0 in warmup stage, type_acc stuck at ~50% (random) |
| 5 | **BatchNorm instability** in 2D spectral frontend | MEDIUM | Running stats may accumulate NaN across epochs |
| 6 | **Curriculum stage transition** optimizer state mismatch | LOW-MEDIUM | Stage warmup may not fully recover from prior dynamics |

## Phase 1: CRITICAL — Fix NaN Explosion (Immediate)

### 1.1 Diagnose NaN Source

**What's happening:** At epoch 5, the product of 5 biquad band frequency responses produces `inf` or `NaN`, which propagates through `hmag_loss` → total_loss → all gradients → model weights.

**Verification:**
```python
# Add to train_one_epoch, after forward pass
for name, val in components.items():
    if not torch.isfinite(val):
        print(f"  [nan] {name} is non-finite at step {self.global_step}")
```

**Likely culprits:**
- `forward_soft` blends coefficients from different filter types → invalid biquad → resonance pole at z=1
- `torch.prod(H_mag_bands, dim=1)` amplifies small numerical errors across 5 bands
- bf16 autocast underflow in `compute_biquad_coeffs_multitype_soft`

### 1.2 Apply NaN-Proofing to DSP Layer

**File: `differentiable_eq.py`**

```python
def forward_soft(self, gain_db, freq, q, type_probs, n_fft=2048):
    # ... existing float32 casting ...
    
    b0, b1, b2, a1, a2 = self.compute_biquad_coeffs_multitype_soft(...)
    
    # ADD: Per-band stability check before frequency response
    a0_check = torch.ones_like(b0)  # already normalized
    pole_check = torch.abs(a1) + torch.abs(a2)
    unstable = pole_check > 1.9  # near unit circle
    if unstable.any():
        # Dampen unstable bands toward flat response
        b0 = torch.where(unstable, torch.ones_like(b0) * 0.5, b0)
        b1 = torch.where(unstable, torch.zeros_like(b1), b1)
        b2 = torch.where(unstable, torch.zeros_like(b2), b2)
        a1 = torch.where(unstable, torch.zeros_like(a1), a1)
        a2 = torch.where(unstable, torch.zeros_like(a2), a2)
    
    H_mag_bands = self.freq_response(b0, b1, b2, a1, a2, n_fft)
    
    # ADD: Per-band clamping BEFORE product
    H_mag_bands = torch.clamp(H_mag_bands, min=1e-6, max=1e3)
    H_mag_total = torch.prod(H_mag_bands, dim=1)
    
    # ADD: Final clamp with gradient-safe bound
    H_mag_total = torch.clamp(H_mag_total, min=1e-6, max=1e4)
    return H_mag_total.to(orig_dtype)
```

### 1.3 Add Gradient Clipping Per-Component

**File: `train.py` — `train_one_epoch`**

The current `clip_grad_norm_(..., max_norm=1.0)` is applied only when accumulation completes. Add per-component NaN detection:

```python
# After loss.backward(), before optimizer.step():
for name, param in self.model.named_parameters():
    if param.grad is not None:
        grad_norm = param.grad.norm()
        if not torch.isfinite(grad_norm):
            param.grad.zero_()  # Zero out NaN gradients
```

## Phase 2: HIGH — Fix Loss Weight Imbalance

### 2.1 Current Loss Component Analysis

From the log (epoch 4, typical step):
```
loss_gain=4.68  loss_freq=1.16  loss_q=0.72  type_loss=1.33
hmag_loss=1.87  contrastive=0.55  spread_loss=-1.38
```

With current weights: `2.0*4.68 + 1.0*1.16 + 1.0*0.72 + 2.0*1.33 + 0.3*1.87 + 0.05*(-1.38) + 0.2*embed_var + 0.05*contrastive`

**Problem:** `loss_gain` contributes ~9.4 dB of the ~15 total loss (63%). The gain head dominates gradient flow, while type learning (2.0 * 1.33 = 2.66) gets drowned out. Type accuracy stays at ~50% (random for 5 classes).

### 3.3 Add Weight Standardization to 2D Convolutions

**File: `model_tcn.py` — `SpectralConvBlock2D`**

```python
def forward(self, x):
    # Weight standardization: normalize conv weights per output channel
    w = self.conv.weight
    w_mean = w.mean(dim=[1,2,3], keepdim=True)
    w_var = w.var(dim=[1,2,3], keepdim=True)
    w_std = torch.sqrt(w_var + 1e-5)
    w_normalized = (w - w_mean) / w_std
    # Use F.conv2d with normalized weights
    x = F.pad(x, (self.time_pad, 0, self.freq_pad, self.freq_pad))
    out = F.conv2d(x, w_normalized, self.conv.bias, 
                   self.conv.stride, 0, self.conv.dilation, self.conv.groups)
    return F.gelu(self.bn(out))
```

### 4.3 Add Type Confidence Regularization

**File: `loss_multitype.py` — add to `MultiTypeEQLoss`**

```python
# After type_loss computation:
# Encourage confident predictions (entropy regularization)
type_probs = pred_type_logits.softmax(-1)
type_entropy = -(type_probs * (type_probs + 1e-8).log()).sum(-1).mean()
loss_type_confidence = type_entropy  # Minimize entropy = maximize confidence
components["type_entropy"] = loss_type_confidence

# Add to total with small weight
```

## Phase 5: MEDIUM — Data Augmentation & Pipeline

### 5.1 Current Augmentation Is Minimal

The dataset only applies random volume scaling (`scale = random.uniform(0.5, 1.0)`). This is insufficient for a 1.8M parameter model.

### 5.2 Add SpecAugment

**File: `train.py` — add to `_prepare_input` or `train_one_epoch`:**

```python
def spec_augment(self, mel_spec, freq_mask_param=20, time_mask_param=40, 
                 num_freq_masks=2, num_time_masks=2):
    """Apply SpecAugment: frequency and time masking."""
    B, n_mels, T = mel_spec.shape
    
    for _ in range(num_freq_masks):
        f = torch.randint(0, freq_mask_param, (1,)).item()
        f0 = torch.randint(0, n_mels - f, (1,)).item()
        mel_spec[:, f0:f0+f, :] = mel_spec.mean(dim=-1, keepdim=True).expand_as(
            mel_spec[:, f0:f0+f, :])
    
    for _ in range(num_time_masks):
        t = torch.randint(0, time_mask_param, (1,)).item()
        t0 = torch.randint(0, T - t, (1,)).item()
        mel_spec[:, :, t0:t0+t] = mel_spec.mean(dim=-2, keepdim=True).expand_as(
            mel_spec[:, :, t0:t0+t])
    
    return mel_spec
```

### 5.3 Add Noise Injection

### 5.4 Verify No Data Leakage

**Check:** The validation dataset is generated independently with `SyntheticEQDataset(size=2000)` and never overlaps with training data. The train/test split uses `random_split` with a fixed seed. **No data leakage detected.**

However, the precomputed cache is generated once at initialization. If the cache is reused across runs with different configs, the data distribution may not match the current training setup.

**Recommendation:** Add a hash of the config to the cache filename:

## Phase 6: LOW-MEDIUM — Learning Rate & Optimizer

### 6.1 Current LR Schedule Analysis

- Base LR: 2e-4 for encoder, 1e-3 for head (5x)
- CosineAnnealing with T_max=80, eta_min=1e-6
- 500-step warmup from 0.1% of base
- Per-stage warmup: 200 steps from 30% to 100%

**Issue:** The head LR (1e-3) may be too aggressive for the fine-grained parameter regression task. The gain head has a `Tanh()` output clamped to [-24, 24] — high LR can cause oscillation at the clamp boundaries.

### 6.2 Recommended LR Adjustments

**File: `conf/config.yaml`**
```yaml
model:
  learning_rate: 0.0001  # Reduce from 2e-4
```

**File: `train.py` — adjust head multiplier:**
```python
param_groups = [
    {"params": encoder_params, "lr": base_lr, "weight_decay": wd, "initial_lr": base_lr},
    {"params": head_params, "lr": base_lr * 3.0, "weight_decay": wd, "initial_lr": base_lr * 3.0},
    # Reduce from 5.0x to 3.0x — head doesn't need 5x the encoder LR
]
```

### 6.3 Switch to ReduceLROnPlateau

CosineAnnealing is schedule-based and doesn't react to validation performance. Switch to plateau-based:

```python
self.scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau(
    self.optimizer, mode='min', factor=0.5, patience=5, min_lr=1e-6
)
# In fit(): call self.scheduler.step(val_loss) instead of .step()
```

### 6.4 Gradient Accumulation: Verify Effective Batch Size

Current: batch_size=32, accumulation=4 → effective batch size = 128.

For 10K training samples (80% train split = 8000 samples): 8000/32 = 250 batches/epoch.

This is adequate. No change needed.

## Implementation Workflow (Iterative)

### Step 1: Fix NaN (Phase 1) — Run first
1. Apply `differentiable_eq.py` NaN-proofing (1.2)
2. Add gradient NaN zeroing (1.3)
3. Run `python train.py` for 10 epochs
4. **Verify:** No NaN in val_loss, train/val both finite

### Step 2: Fix Loss Weights (Phase 2) — If NaN fixed
1. Update `conf/config.yaml` loss weights (2.2)
2. Add loss component clamping (2.3)
3. Run for 20 epochs
4. **Verify:** Type accuracy > 60%, loss components balanced

### Step 3: Add Regularization (Phase 3) — If loss balanced
1. Increase weight_decay to 0.01
2. Increase dropout to 0.2
3. Run for 30 epochs
4. **Verify:** Train/val gap < 10% of val loss

### Step 4: Tune Gumbel & LR (Phases 4, 6) — If regularization helps
1. Update curriculum temperature schedule
2. Reduce base LR to 1e-4
3. Run for full 80 epochs
4. **Verify:** Converged val_loss < 10, type_acc > 70%

### Step 5: Data Augmentation (Phase 5) — If still overfitting
1. Add SpecAugment
2. Add noise injection
3. Increase dataset to 30K if needed
4. **Verify:** Train/val gap eliminated

1. **GPU memory utilization:** Is the model memory-bound? If yes, gradient checkpointing could be enabled to allow larger batch sizes.

2. **Previous training runs:** What were the final metrics for v6, v7, v9, v10, v11, v12? The log files exist but haven't been analyzed.

3. **Validation metric targets:** What are acceptable MAE thresholds for gain (dB), frequency (octaves), Q (decades), and type accuracy (%)?

4. **Inference requirements:** Does the model need to run in real-time streaming mode? This affects whether we can increase model capacity.

5. **Real audio evaluation:** Has `test_real_audio.py` been run against the best checkpoint? Synthetic-to-real gap may be a bigger issue than train/val gap.

6. **The `training_v8.log` shows `train=0.0000` from epoch 6** — this means the NaN recovery mechanism (`_recover_from_nan`) is failing to find any clean checkpoint. The corruption starts earlier than detected. Consider adding NaN checks *during* training (per-batch) rather than post-epoch.
